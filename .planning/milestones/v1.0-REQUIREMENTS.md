# Requirements: KDE Fan Control

**Defined:** 2026-04-10
**Core Value:** Users can safely and flexibly control desktop fan behavior with understandable per-fan PID policies, without losing fail-safe behavior.

## v1 Requirements

Requirements for initial release. Each maps to roadmap phases.

### Hardware Discovery

- [x] **HW-01**: User can list detected temperature sensors exposed by supported Linux hardware interfaces
- [x] **HW-02**: User can list detected fan controllers and fan channels exposed by supported Linux hardware interfaces
- [x] **HW-03**: User can inspect capabilities for each fan channel, including whether PWM control or voltage control is available
- [x] **HW-04**: User can assign and persist a friendly name for a detected temperature sensor
- [x] **HW-05**: User can assign and persist a friendly name for a detected fan controller or fan channel
- [x] **HW-06**: User can see whether each discovered fan or control endpoint is `available`, `partial`, or `unavailable`
- [x] **HW-07**: User can inspect the reason a discovered endpoint is partially supported or unavailable
- [x] **HW-08**: User can inspect whether a fan channel exposes tach or RPM feedback

### Fan Enrollment And Modes

- [x] **FAN-01**: User can leave a detected fan under BIOS or existing system control without daemon interference
- [x] **FAN-02**: User can enroll a detected fan for daemon-managed control when safe managed control is supported
- [x] **FAN-03**: User cannot enroll hardware that lacks enough control or fallback support for safe daemon management
- [x] **FAN-04**: User can choose the hardware control mode used by the daemon when the hardware exposes multiple modes such as PWM or voltage
- [x] **FAN-05**: User can view whether a fan is unmanaged, managed, fallback, partial, or unavailable
- [x] **FAN-06**: User can reboot the system and have previously managed fans resume daemon control automatically from persisted configuration

### Sensor Sources And Aggregation

- [x] **SNS-01**: User can select a single temperature sensor as the control input for a fan
- [x] **SNS-02**: User can select multiple temperature sensors as the control input group for a fan
- [x] **SNS-03**: User can choose `average` as the aggregation function for a multi-sensor group
- [x] **SNS-04**: User can choose `max` as the aggregation function for a multi-sensor group
- [x] **SNS-05**: User can choose `min` as the aggregation function for a multi-sensor group
- [x] **SNS-06**: User can choose `median` as the aggregation function for a multi-sensor group

### PID Control

- [x] **PID-01**: User can set a target temperature for each managed fan
- [x] **PID-02**: User can configure P, I, and D gains for each managed fan
- [x] **PID-03**: Daemon computes fan output continuously from the selected sensor input or aggregation and target temperature
- [x] **PID-04**: Daemon applies output within safe supported bounds for the selected hardware control mode
- [x] **PID-05**: User can enable basic PID auto-tuning for a managed fan
- [x] **PID-06**: User can inspect the resulting tuned P, I, and D values after auto-tuning
- [x] **PID-07**: User can understand that v1 managed control is thermal-control PID based on temperature input, not RPM-target tracking

### Safety And Runtime Behavior

- [x] **SAFE-01**: Daemon restores or drives previously daemon-controlled fans to high speed when the service fails or exits unexpectedly
- [x] **SAFE-02**: Daemon does not modify unmanaged fans during normal operation, startup, shutdown, or failure handling
- [x] **SAFE-03**: User can inspect whether a fan is currently in a safe fallback state
- [x] **SAFE-04**: Daemon rejects invalid configurations that would leave a managed fan without a usable temperature input or target
- [x] **SAFE-05**: User can enroll a fan with writable control even if RPM feedback is unavailable
- [x] **SAFE-06**: Daemon safety logic does not depend on tach presence to force safe-maximum output for managed fans
- [x] **SAFE-07**: If startup cannot safely apply persisted control to a managed fan, the daemon surfaces a degraded state instead of silently assuming success

### DBus API

- [x] **BUS-01**: User-space clients can query discovered hardware and capabilities over DBus
- [x] **BUS-02**: User-space clients can create, update, and delete fan-control configuration over DBus
- [x] **BUS-03**: User-space clients can read current runtime status for sensors, fans, and control policies over DBus
- [x] **BUS-04**: User-space clients can trigger persistence of configuration through the daemon over DBus
- [x] **BUS-05**: User-space clients can start auto-tuning through DBus
- [x] **BUS-06**: DBus clients observe the daemon as the sole authority for discovery, persistence, and runtime state

### Persistence

- [x] **CONF-01**: Daemon persists exactly one active configuration in v1
- [x] **CONF-02**: Persisted configuration stores friendly names, enrollment state, selected sensor inputs, aggregation settings, control mode, target temperature, and PID parameters
- [x] **CONF-03**: Persisted configuration is validated against current discovered hardware before control is resumed on boot

### CLI

- [x] **CLI-01**: User can list sensors, fans, capabilities, and support state from the CLI
- [x] **CLI-02**: User can configure friendly names, fan enrollment, sensor inputs, aggregation function, and PID settings from the CLI
- [x] **CLI-03**: User can inspect current fan-control status, active temperatures, and fault state from the CLI
- [x] **CLI-04**: User can trigger PID auto-tuning from the CLI

### KDE GUI

- [x] **GUI-01**: User can view discovered sensors, fans, support state, and current status in a Qt6 or QML GUI
- [x] **GUI-02**: User can configure fan enrollment, temperature inputs, aggregation function, target temperature, control mode, and PID settings in the GUI
- [x] **GUI-03**: User can trigger auto-tuning from the GUI
- [x] **GUI-04**: User can access current status from a system tray icon
- [x] **GUI-05**: User can recognize unmanaged fans versus daemon-controlled fans and unsupported hardware in the GUI

## v2 Requirements

Deferred to future release. Tracked but not in current roadmap.

### Profiles

- **PROF-01**: User can store multiple named fan-control profiles
- **PROF-02**: User can switch between profiles safely

### Advanced Control

- **CTRL-01**: User can define additional statistical aggregation functions beyond average, max, min, and median
- **CTRL-02**: User can define weighted sensor combinations
- **CTRL-03**: User can use adaptive or fuzzy tuning strategies when proven safe on supported hardware

### Observability

- **OBS-01**: User can view historical sensor and fan graphs
- **OBS-02**: User can export diagnostic snapshots for troubleshooting

### Compatibility Expansion

- **COMP-01**: User can use richer vendor-specific integrations for supported hardware families
- **COMP-02**: User can receive compatibility guidance for unsupported or partially supported controllers

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Web UI | Local desktop management is the priority |
| Non-KDE polished parity in v1 | KDE and Qt6-first reduces UI surface area and design overhead |
| Remote multi-machine control | Not required for the desktop local-control use case |
| User-authored custom formulas or plugins in v1 | Adds complexity before core control and safety are proven |
| GPU, AIO, and embedded-controller breadth-first support | v1 should focus on a trustworthy generic hwmon or sysfs desktop path |
