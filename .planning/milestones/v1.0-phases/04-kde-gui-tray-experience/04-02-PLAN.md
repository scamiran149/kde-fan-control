Create the C++ models needed for fan detail editing, validation, and auto-tune flow.

**1. DraftModel** (`gui/src/models/draft_model.h` and `.cpp`):
- Class `DraftModel : public QObject` with `QML_ELEMENT`
- Q_PROPERTY: `QString fanId` — the currently selected fan
- Q_PROPERTY: `bool enrolled` — local enrollment state (editable)
- Q_PROPERTY: `QString controlMode` — editable (e.g., "pwm"/"voltage")
- Q_PROPERTY: `QStringList sensorIds` — editable sensor source list
- Q_PROPERTY: `QString aggregation` — editable ("average"/"max"/"min"/"median")
- Q_PROPERTY: `double targetTempCelsius` — editable target (converted from millidegrees for display)
- Q_PROPERTY: `double kp` — editable proportional gain
- Q_PROPERTY: `double ki` — editable integral gain
- Q_PROPERTY: `double kd` — editable derivative gain
- Q_PROPERTY: `bool autoTuneRunning` — tracks whether auto-tune is active for this fan
- Q_PROPERTY: `bool autoTuneProposalAvailable` — true when an auto-tune proposal is pending
- Q_PROPERTY: `double proposedKp`, `proposedKi`, `proposedKd` — auto-tune proposed gains
- Q_PROPERTY: `bool hasValidationError` — true when last validation found errors
- Q_PROPERTY: `QStringList validationErrors` — per-fan validation error strings
- Q_PROPERTY: `bool hasApplyError` — true when last apply partially failed
- Q_PROPERTY: `QStringList applyErrors` — per-fan apply error strings
- Q_INVOKABLE `void loadFan(const QString &fanId)` — fetches draft config and applied config for this fan from DaemonInterface, merges them (applied values as baseline, draft overrides if present), populates all Q_PROPERTY fields
- Q_INVOKABLE `void setEnrolled(bool enrolled)` — updates local state, calls `DaemonInterface.setDraftFanEnrollment(fanId, json)` with enrollment change
- Q_INVOKABLE `void setSensorIds(const QStringList &sensorIds)` — updates local state, calls `DaemonInterface.setDraftFanControlProfile(fanId, json)` with sensor change
- Q_INVOKABLE `void setAggregation(const QString &agg)` — similar profile update
- Q_INVOKABLE `void setTargetTempCelsius(double celsius)` — converts to millidegrees, sends profile update
- Q_INVOKABLE `void setPidGains(double kp, double ki, double kd)` — sends profile update
- Q_INVOKABLE `void setControlMode(const QString &mode)` — sends enrollment update with mode change
- Q_INVOKABLE `void validateDraft()` — calls DaemonInterface.validateDraft(), parses result JSON, sets hasValidationError and validationErrors
- Q_INVOKABLE `void applyDraft()` — calls DaemonInterface.applyDraft(), parses result JSON, sets hasApplyError and applyErrors. If partially applied, shows warning banner.
- Q_INVOKABLE `void discardDraft()` — calls DaemonInterface.discardDraft(), then reloads fan data
- Q_INVOKABLE `void startAutoTune()` — calls DaemonInterface.startAutoTune(fanId), sets autoTuneRunning = true
- Q_INVOKABLE `void acceptAutoTuneProposal()` — calls DaemonInterface.acceptAutoTune(fanId), stages proposed gains into local draft, then user must Apply
- Q_INVOKABLE `void dismissAutoTuneProposal()` — clears autoTuneProposalAvailable, discards proposal
- Connect to StatusMonitor's signal for `AutoTuneCompleted` → when fanId matches, call DaemonInterface.autoTuneResult(fanId) to get proposed gains, set autoTuneProposalAvailable = true, populate proposedKp/Ki/Kd
- Connect to StatusMonitor's `draft_changed` signal → reload fan data if current fan matches
- Connect to StatusMonitor's `control_status_changed` signal → refresh runtime status

- Each event: `{ "timestamp": "...", "event_type": "...", "reason": "...", "detail": "...", "fan_id": "..." }`
- Sorted by timestamp descending (most recent first)

**1. FanDetailPage.qml** — `Kirigami.ScrollablePage`:
- Title: fan display name from FanListModel
- Pushed onto `pageStack` from Overview or Inventory when user selects a fan (per D-01/D-13)

**Header block** (always visible at top, per UI-SPEC):
- Fan name (from FanListModel.displayName)
- State badge (StateBadge component from Plan 01)
- Managed/unmanaged support summary text
- Current aggregated temperature (TemperatureDisplay component)
- RPM label (integer or "No RPM feedback")
- Output percent bar (OutputBar component)
- High-temp alert pill when `fanState.highTempAlert === true` (separate from base state per UI-SPEC)

**Core controls** (visible first, before tabs, per D-13/D-14):
- Use `Kirigami.FormLayout` for edit fields
- Top-to-bottom order per UI-SPEC:
  1. **Enrollment state toggle**: `Controls.CheckBox` "Managed" — calls `draftModel.setEnrolled(checked)` and flushes draft
  2. **Control mode selector**: `Controls.ComboBox` with model ["PWM"] (or ["PWM", "Voltage"] when available from fan's control_modes) — calls `draftModel.setControlMode(currentText.toLowerCase())`
  3. **Temperature source selector**: Multi-select sensor list — `Controls.CheckBox` per sensor, bound to `draftModel.sensorIds`. Per UI-SPEC: if exactly 1 sensor selected, show read-only text "Single sensor" instead of aggregation dropdown
  4. **Aggregation selector**: `Controls.ComboBox` with model ["Average", "Max", "Min", "Median"] — calls `draftModel.setAggregation(currentText.toLowerCase())`. Visible only when 2+ sensors selected (per UI-SPEC). Hidden and shows "Single sensor" text when 1 sensor
  5. **Target temperature field**: `Controls.SpinBox` or `Controls.TextField` with validator — calls `draftModel.setTargetTempCelsius(value)`. Suffix "°C". Accept decimals to 1 place
  6. **PID gains**: Three `PidField` components for Kp, Ki, Kd — each bound to DraftModel properties. Per D-15, include hover explanations
  7. **Start auto-tune action**: `Kirigami.Action` or `Controls.Button` "Start auto-tune" — calls `draftModel.startAutoTune()`. Disabled when `draftModel.autoTuneRunning === true` (per D-17, per UI-SPEC "disable a second start action for that fan")

**Draft editing actions** (per D-03/D-21):
- Three action buttons in order per UI-SPEC: "Validate draft", "Apply changes", "Discard draft"
- "Validate draft" — calls `draftModel.validateDraft()`, shows `Kirigami.InlineMessage`:
  - Success: "Valid: {N} enrollable entries"
  - Warning/Error: "{N} rejected entries" with per-field error text
  - Per-field error text for the active fan shown inline below the relevant field
  - Validation error messages use simplified backend language per UI-SPEC: "no target temperature", "no temperature source", "unsupported control mode", "invalid cadence", etc.
- "Apply changes" — calls `draftModel.applyDraft()`, shows:
  - Best-effort partial apply warning banner if any fan rejected (per UI-SPEC: "never show a success toast if any fan was rejected")
  - Success banner only if all fans applied
- "Discard draft" — calls `draftModel.discardDraft()`, shows confirmation dialog per UI-SPEC copy: "Discard all staged changes? This keeps the current applied configuration and removes the current draft."

**Auto-tune completion banner** (per D-18/D-19):
- When `draftModel.autoTuneProposalAvailable === true`, show `Kirigami.InlineMessage`:
  - Heading: "Auto-tune finished"
  - Body: "Review the proposed PID gains before applying them."
  - Actions: "Accept proposed gains" (calls `draftModel.acceptAutoTuneProposal()`) and "Dismiss proposal" (calls `draftModel.dismissAutoTuneProposal()`)
- Accepted gains are staged into draft only — user must still click "Apply changes" per D-19
- Auto-tune failure shows inline error: "Auto-tune could not complete. Check sensor availability and current runtime state, then try again."

**Advanced tabs** (per D-16, using `Controls.TabBar` + `StackLayout`):
1. **Runtime tab**:
   - Live sensor IDs/names used (from runtime status)
   - Aggregation mode (from runtime status)
   - Target temperature (from applied config, °C)
   - Current error (difference between actual and target, in °C)
   - Mapped PWM value when present
   - Auto-tune state (idle / running / completed / failed)
   - High-temp alert state (bool)

2. **Advanced tab**:
   - Cadence: sample_interval_ms, control_interval_ms, write_interval_ms (SpinBox with ms suffix)
   - Deadband: deadband_millidegrees (SpinBox, displayed as °C with 1 decimal)
   - Actuator policy: output_min_percent, output_max_percent (SpinBox)
   - PID limits: output_min_percent, output_max_percent (SpinBox)

3. **Events tab**:
   - `ListView` with `model: lifecycleEventModel` showing timestamp, event type, reason, fan ID
   - Sorted most recent first

**2. PidField.qml** — Reusable PID gain input component:
- Properties: `label` (string), `value` (double), `helpText` (string), `onValueChanged` (callback)
- Visual: `Kirigami.FormData.label` for the name, `Controls.SpinBox` with `from: 0`, `to: 100`, `stepSize: 0.1`, `decimals: 2`
- Hover explanation via `Controls.ToolTip.visible: hovered` per D-15
- Predefined help texts per UI-SPEC:
  - Kp: "Responds to the current temperature gap. Higher values react faster but can hunt."
  - Ki: "Corrects steady offset over time. Higher values remove drift but can overshoot."
  - Kd: "Damps fast temperature swings. Higher values can reduce overshoot but may amplify noise."

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GUI → DBus → Daemon (writes) | Draft writes and auto-tune triggers cross privilege boundary; daemon requires UID 0 |
| Auto-tune proposal | Proposals are untrusted until user explicitly accepts and applies |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Elevation of Privilege | Draft write methods | mitigate | When UID-0 authorization fails, show inline error per UI-SPEC: "This action requires elevated privileges. Run the GUI with the required authorization and try again." Never silently skip or cache write permissions. |
| T-04-06 | Tampering | Auto-tune proposal | mitigate | Auto-tune proposed gains are staged into draft only, not directly applied. User must review and explicitly "Apply changes" per D-19. Banner clearly states "Review the proposed PID gains before applying them." |
| T-04-07 | Tampering | Validation bypass | mitigate | "Validate draft" is offered separately from "Apply changes" per UI-SPEC contract. Validation errors are displayed prominently with per-field detail. User cannot bypass validation — apply calls validate internally on the daemon side. |
</threat_model>
