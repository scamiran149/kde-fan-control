### UAT 08-01: Inventory & Naming

**Precondition:** Daemon running with at least one sensor and one fan discovered. GUI launched and connected.

1. Open the GUI -> Overview page shows fan cards with state badges, temperatures, and output bars
2. Navigate to Inventory page via GlobalDrawer -> verify sensor list shows device names and temperatures
3. Verify fan list shows fan names and support states (unmanaged/managed/partial/unavailable)
4. On a fan card in Overview, verify: state badge color/text, temperature display, RPM (if tach sensor present), output percentage bar
5. Test friendly naming: edit a sensor or fan display name -> verify the name persists after the 3-second polling refresh
6. Verify temperatures update at least every 3 seconds while the window is visible

### UAT 08-02: Draft & Apply Cycle

**Precondition:** Daemon running, at least one unmanaged fan available.

1. Click an unmanaged fan row in Overview -> FanDetailPage opens showing fan details
2. Toggle "Enroll this fan" ON -> fan enters draft enrollment state
3. Select control mode (PWM or Voltage, depending on fan capability)
4. Select one or more temperature sensor sources
5. If multiple sensors selected, choose aggregation function (mean, max, min)
6. Set target temperature (e.g., 50.0 degrees Celsius)
7. Adjust PID gains (Kp, Ki, Kd) using the PidField SpinBoxes
8. Click "Validate Draft" -> verify validation result appears (success message or inline error)
9. If validation succeeds, click "Apply Draft" -> verify:
   - Fan transitions from "unmanaged" to "managed" state in Overview
   - Output bar shows the fan being actively controlled
   - Temperature reflects real-time sensor data
10. Click "Discard Draft" on a different fan -> verify draft is discarded without applying changes

### UAT 08-03: Tray & Notifications

**Precondition:** GUI running with tray icon visible.

1. Verify tray icon shows current worst severity (managed -> emblem-ok-symbolic or equivalent)
2. Hover over tray icon -> verify tooltip shows severity summary and managed/alert counts
3. Click tray icon -> verify main window appears/raises
4. Minimize main window -> verify tray icon stays present and polling pauses
5. Restore window from tray -> verify polling resumes and data refreshes
6. Trigger a state change (e.g., introduce a condition that causes degraded state) -> verify:
   - Desktop notification appears for degraded transition (only on transition, not sustained)
   - Tray icon changes to data-warning-symbolic
   - Severity banner appears in Overview page
7. Click "Acknowledge alerts" in tray context menu or tray popover -> verify sticky alert banners clear in the UI

### UAT 08-04: Auto-Tune Flow

**Precondition:** A managed fan with sensor sources configured.

1. Open FanDetailPage for a managed fan
2. Click "Start Auto-Tune" button -> verify:
   - Button disables immediately
   - autoTuneRunning property becomes true
   - UI shows "Auto-tune in progress" indication
3. Wait for auto-tune completion -> verify:
   - Proposal banner appears in the detail page with the proposed PID gains
   - Accept/dismiss buttons are visible
4. Click "Accept proposed gains" -> verify:
   - Gains stage into draft PID fields
   - User can still validate and apply
5. Validate and apply the draft -> verify fan uses the new gains
6. Alternatively, verify "Dismiss proposal" discards the auto-tune results

### UAT 08-05: Reactive DBus Updates

**Precondition:** GUI running with daemon connected. Overview showing live fan data.

This test verifies that the DBus signal wiring from Plan 06 works correctly — data should update without requiring user interaction.

1. From another terminal, use the CLI to change a fan's friendly name:
   `kde-fan-control-cli set-fan-name <fan-id> "Test Name"`
2. Verify the GUI overview updates to show the new name within 5 seconds (3s polling + signal propagation)
3. From the CLI, apply a configuration change that affects fan control:
   `kde-fan-control-cli apply-draft`
4. Verify the GUI reflects the change (draft_changed signal -> re-fetch -> model update)
5. If possible, trigger a degraded or fallback condition and verify:
   - The degraded_state_changed signal fires
   - GUI updates severity badges and banners in real time
   - A desktop notification appears on the transition
6. Verify no QML binding loop warnings or excessive DBus traffic in console output

### UAT 08-06: Window Visibility and Polling Pause

**Precondition:** GUI running with daemon connected, showing live data.

1. Observe console output or DBus traffic (using `busctl monitor` on the system bus)
2. Verify poll requests appear approximately every 3 seconds
3. Minimize the window -> verify polling stops (no new requests for 10+ seconds)
4. Restore the window -> verify polling resumes
5. Close to tray (if applicable) -> verify polling stays paused
6. Reopen from tray -> verify polling resumes immediately with a full refresh
