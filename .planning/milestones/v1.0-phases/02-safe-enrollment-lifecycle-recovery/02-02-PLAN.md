# Plan 02-02: DBus Draft-Apply Contract And Authorization Boundary

**Phase:** 02-safe-enrollment-lifecycle-recovery
**Status:** Ready
**Goal:** Expose daemon-owned lifecycle configuration over DBus with staged edits, explicit apply, read-open/write-privileged access, and signals for lifecycle changes.

## Requirements Addressed

- BUS-02
- BUS-04
- BUS-06
- CLI-02
- CONF-01
- CONF-02

## Why This Plan Exists

The Phase 2 user model depends on draft edits with explicit apply. That requires a DBus contract that distinguishes read-only status from privileged mutation paths and that keeps authorization at the daemon boundary so future `polkit` support can slot in without changing the API shape.

## Scope

- Add DBus methods for draft lifecycle configuration
- Add DBus methods for apply, discard, and validation feedback
- Add inspectable read methods for draft, applied, degraded, and event state
- Add lifecycle signals so clients do not rely only on polling
- Introduce a write-authorization seam that currently supports privileged callers and later `polkit`

## Tasks

1. Define the DBus lifecycle surface in `crates/daemon/src/main.rs`
- Add methods to read applied config, draft config, degraded summary, and recent lifecycle events
- Add methods to stage enrollment changes and control-mode changes
- Add methods to validate draft, apply draft, and discard draft

2. Introduce lifecycle change signals
- Signal when draft changes
- Signal when applied config changes
- Signal when degraded or fallback state changes
- Signal when lifecycle event history is appended

3. Add authorization boundary
- Keep read methods accessible to local users
- Gate write methods through a dedicated authorization helper
- Implement initial policy as privileged-caller-only without embedding `sudo` assumptions into the CLI
- Structure the helper so a future `polkit` implementation can replace or wrap it without changing the DBus method contract

4. Normalize payloads
- Replace the current inventory-as-JSON-string pattern where needed with phase-appropriate payload conventions, but keep changes minimal and compatible with the existing client pattern unless typed payloads are clearly worth the extra surface
- Ensure error responses are explicit and user-actionable for invalid draft edits or refused enrollment

## Files To Modify

- `crates/daemon/src/main.rs`
- `crates/cli/src/main.rs`
- `crates/core/src/config.rs`
- add shared lifecycle model modules in `crates/core/src/` if needed

## Design Constraints

- DBus remains the only configuration mutation boundary
- Authorization policy must be separable from business logic
- Draft changes must not touch live ownership until apply succeeds
- The API should remain stable across the later transition to package-installed `polkit`-mediated writes

<threat_model>
## Threat Model

- **Threat:** unprivileged callers mutate daemon-owned lifecycle state
  - **Mitigation:** gate all write methods through explicit authorization checks at the daemon boundary
- **Threat:** clients depend on CLI-specific auth behavior that breaks later GUI or `polkit` flows
  - **Mitigation:** keep CLI as a thin DBus client and avoid client-side privilege assumptions
- **Threat:** clients miss important state transitions and act on stale polling results
  - **Mitigation:** emit lifecycle signals for applied, degraded, and fallback changes
</threat_model>

## Verification

3. Confirm write methods fail cleanly for unprivileged callers
4. Confirm draft changes are visible through read methods before apply

## Done When

- Clients can inspect draft, applied, degraded, and event state over DBus
- Clients can stage and apply lifecycle changes through daemon-owned methods
- Write authorization is enforced at the daemon boundary
- The DBus contract is compatible with later `polkit` introduction without changing method semantics
