# Plan 02-03: Boot Reconciliation, Ownership Tracking, And Fallback Lifecycle

**Phase:** 02-safe-enrollment-lifecycle-recovery
**Status:** Ready
**Goal:** Implement safe startup reconciliation, per-fan best-effort restore, daemon ownership tracking, and crash-path fallback so managed fans behave safely across boot and daemon failure.

## Why This Plan Exists

Phase 2’s highest-risk behavior is lifecycle correctness: the daemon must restore managed fans on boot when safe, refuse unsafe restore, preserve unmanaged fans, and drive previously managed fans to safe maximum on failure or shutdown paths.

## Scope

- Reconcile persisted applied config against live inventory at startup
- Restore safe matches as managed
- Skip mismatched or unsafe fans and surface degraded state with reasons
- Track exactly which fans the daemon owns at runtime
- Force safe maximum output for previously managed fans on failure or shutdown

## Tasks

1. Build startup reconciliation logic
- On daemon startup, load applied config and current inventory
- For each applied-managed fan, verify stable ID match and safe capability match
- Restore matching fans as managed
- Skip missing or now-unsafe fans with explicit degraded reasons
- Record lifecycle events for restore success, skip, and mismatch cases

2. Track runtime ownership explicitly
- Create a runtime-owned set of fans the daemon has successfully claimed
- Ensure unmanaged fans are never inserted into that set
- Use this set as the authority for fallback fan writes

3. Implement safe-maximum fallback lifecycle
- Add code path to force safe maximum output for owned fans during daemon stop or failure handling
- Keep fallback semantics separate from degraded startup mismatch semantics
- Ensure no fallback writes are attempted for unmanaged fans or fans never successfully claimed

4. Expose inspectable runtime state
- Surface which fans are currently managed, degraded, or in fallback
- Surface degraded reasons and recent lifecycle events through the DBus status model

## Files To Modify

- `crates/daemon/src/main.rs`
- `crates/core/src/inventory.rs` if capability helpers need extension
- `crates/core/src/config.rs`
- new daemon runtime modules if the lifecycle logic becomes too large for `main.rs`

## Design Constraints

- Startup behavior is best-effort partial apply, not fail-whole-service
- Unmanaged fans remain untouched in startup, runtime, and fallback paths
- Writable control without tach remains valid if safe-maximum fallback is possible
- Degraded startup and active fallback must be represented as distinct states

<threat_model>
## Threat Model

- **Threat:** daemon writes fallback values to unmanaged fans
  - **Mitigation:** fallback writes use an explicit owned-fan set populated only after successful claim
- **Threat:** stale config restores control onto a different physical fan after hardware changes
  - **Mitigation:** reconcile against stable IDs and current safe capabilities before claiming ownership
- **Threat:** startup silently skips unsafe restore and misleads the user
  - **Mitigation:** record degraded state and lifecycle events for every skipped or mismatched restore
</threat_model>

## Verification

1. `cargo check`
2. Add tests for reconciliation outcomes:
   - exact match restore
   - missing fan ID
   - changed support state
   - changed control mode support
3. Simulate shutdown/failure paths and confirm only owned fans receive safe-maximum writes

## Done When

- Boot restoration works on a per-fan best-effort basis
- Unsafe or missing fans are skipped with explicit degraded reasons
- Runtime ownership is tracked separately from mere configuration intent
- Crash or stop lifecycle can force safe maximum output for owned fans only
