# Plan 02-01: Managed Config Domain And Persistence

**Phase:** 02-safe-enrollment-lifecycle-recovery
**Status:** Ready
**Goal:** Define and implement the daemon-owned lifecycle configuration model so managed fan ownership, draft edits, applied config, and degraded-state persistence all have a stable foundation.

## Why This Plan Exists

Phase 1 only persists friendly names. Phase 2 needs a real daemon-owned lifecycle config that can represent draft edits, applied configuration, managed fan policy, and degraded-state context without mixing those concerns into the live inventory model.

## Scope

- Extend the config domain to support lifecycle state for fans
- Model draft vs applied configuration explicitly
- Persist the single active authoritative applied config
- Store enough degraded-state context to explain boot mismatch or failed apply outcomes
- Keep the model ready for future Phase 3 live setpoint adjustment without redesigning the storage boundary

## Tasks

1. Extend `crates/core/src/config.rs`
- Add lifecycle config structures for draft and applied config
- Represent fan enrollment by stable fan ID from Phase 1
- Add selected control mode for managed fans
- Add versioned top-level config structure so future schema changes are manageable

2. Define validation helpers in core
- Validate referenced fan IDs exist in current inventory
- Validate selected control mode is supported by that fan
- Reject unsupported or unsafe hardware from applied management state
- Keep validation independent from DBus so both startup reconciliation and apply flows can reuse it

3. Model degraded-state data
- Add structured mismatch or degraded reason types
- Add a small bounded lifecycle event representation with timestamp and reason text
- Separate persisted applied config from ephemeral runtime state while keeping degraded reasons serializable for inspection

4. Preserve existing naming behavior
- Keep friendly-name persistence intact
- Fold names into the expanded top-level config cleanly rather than leaving parallel persistence logic around

## Files To Modify

- `crates/core/src/config.rs`
- `crates/core/src/lib.rs`
- `crates/core/src/inventory.rs` if shared validation helpers or capability flags need minor extension

## Design Constraints

- Exactly one applied configuration in v1
- Draft config must not implicitly become live config
- Config references must use stable fan IDs, not labels or raw sysfs paths
- Unsafe hardware remains visible but cannot be marked as applied-managed

<threat_model>
## Threat Model

- **Threat:** invalid or stale persisted config claims the wrong fan after hardware change
  - **Mitigation:** store stable fan IDs and require reconciliation against current inventory before activation
- **Threat:** malformed config bypasses enrollment safety checks
  - **Mitigation:** centralize validation helpers in core and reuse them from DBus apply and boot startup paths
- **Threat:** future write callers mutate applied config directly
  - **Mitigation:** keep explicit draft vs applied data structures and disallow write-through shortcuts
</threat_model>

## Verification

1. `cargo check`
2. Add targeted unit tests for config serialization and validation rules
3. Round-trip config save and load with:
   - friendly names only
   - managed fan entries
   - invalid control-mode selection
   - stale fan ID references
4. Confirm draft and applied config are stored distinctly

## Done When

- The daemon has a versioned config model that can represent draft and applied lifecycle state
- Validation helpers reject unsupported fan enrollment and invalid mode selection
- Existing friendly-name persistence still works under the expanded config schema
- Future plans can build DBus apply and startup reconciliation on top of this model without redefining persistence
