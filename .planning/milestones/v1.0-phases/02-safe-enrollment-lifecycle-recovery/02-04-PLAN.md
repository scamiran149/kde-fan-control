# Plan 02-04: CLI Lifecycle Flows And Inspectable Recovery State

**Phase:** 02-safe-enrollment-lifecycle-recovery
**Status:** Ready
**Goal:** Provide complete DBus-backed CLI flows for staged enrollment, apply or discard, control-mode selection, and inspection of degraded or fallback lifecycle state.

## Requirements Addressed

- FAN-01
- FAN-02
- FAN-03
- FAN-04
- FAN-05
- CLI-02
- SAFE-03

## Why This Plan Exists

Phase 2 is only useful if users can actually manage lifecycle policy from the CLI and understand the daemon’s recovery decisions. The CLI must make the staged/apply model obvious and surface degraded or fallback state in plain language.

## Scope

- Add CLI commands for staged enrollment and control-mode selection
- Add CLI commands to inspect draft vs applied config
- Add CLI commands to apply or discard draft changes
- Add CLI output for degraded state, fallback state, and recent lifecycle events
- Keep all lifecycle writes DBus-backed and authorization-neutral from the client perspective

## Tasks

1. Extend CLI subcommands
- Add commands to stage managed or unmanaged state for a fan
- Add command to stage the selected control mode for an enrollable fan
- Add commands to show draft config and applied config
- Add commands to validate, apply, and discard draft state

2. Improve inspectability
- Add CLI output for current lifecycle state per fan
- Add CLI output for degraded reasons and recent lifecycle events
- Make skipped boot restore and refused enrollment reasons obvious in text output

3. Preserve thin-client architecture
- Reuse DBus methods rather than reimplementing validation or config logic in the CLI
- Keep existing fallback-to-direct inventory read behavior for inventory-only commands where appropriate, but lifecycle mutation must remain daemon-owned

4. Keep the future UX path open
- Present lifecycle state and event output in a way that can map naturally into Phase 4 GUI and tray concepts later

## Files To Modify

- `crates/cli/src/main.rs`
- shared display helpers if needed in `crates/core/src/`

## Design Constraints

- CLI must reflect staged vs applied state clearly
- CLI must not imply production `sudo` as the intended long-term UX
- Error output must distinguish:
  - unsupported enrollment
  - unauthorized write attempt
  - degraded boot restore
  - active fallback

<threat_model>
## Threat Model

- **Threat:** CLI output hides degraded or fallback state, leading users to trust an unsafe system state
  - **Mitigation:** make lifecycle state, degraded reasons, and event history explicit in command output
- **Threat:** CLI grows its own lifecycle logic and diverges from daemon behavior
  - **Mitigation:** keep validation and mutation daemon-owned over DBus
- **Threat:** users assume immediate write-through despite staged config semantics
  - **Mitigation:** use explicit CLI commands and output for draft, validate, apply, and discard
</threat_model>

## Verification

1. `cargo check`
2. Verify CLI can stage changes without applying them
3. Verify apply and discard behave distinctly
4. Verify control-mode selection refuses unsupported modes cleanly
5. Verify degraded and fallback state are visible through normal CLI inspection flows

## Done When

- Users can manage Phase 2 lifecycle behavior entirely from DBus-backed CLI flows
- Staged vs applied config is obvious from command output
- Degraded and fallback states are inspectable without reading daemon logs
- The CLI remains a thin client over the daemon’s authoritative lifecycle contract
