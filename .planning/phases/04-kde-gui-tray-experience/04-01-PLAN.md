**3. DaemonInterface** (`gui/src/daemon_interface.h` and `.cpp`):
- Class `DaemonInterface : public QObject` — central C++ DBus proxy
- Q_PROPERTY: `bool connected` (NOTIFY connectedChanged)
- Q_PROPERTY: `QString lastError` (NOTIFY lastErrorChanged)
- Private `QDBusInterface` instances for each of the three daemon interfaces: Inventory at `/org/kde/FanControl`, Lifecycle at `/org/kde/FanControl`, Control at `/org/kde/FanControl`
- All methods are Q_INVOKABLE and call daemon via asyncCall on the system bus to `org.kde.FanControl`
- Read methods:
  - `QString snapshot()` — calls `org.kde.FanControl.Inventory.snapshot()`
  - `QString draftConfig()` — calls `org.kde.FanControl.Lifecycle.get_draft_config()`
  - `QString appliedConfig()` — calls `org.kde.FanControl.Lifecycle.get_applied_config()`
  - `QString degradedSummary()` — calls `org.kde.FanControl.Lifecycle.get_degraded_summary()`
  - `QString lifecycleEvents()` — calls `org.kde.FanControl.Lifecycle.get_lifecycle_events()`
  - `QString runtimeState()` — calls `org.kde.FanControl.Lifecycle.get_runtime_state()`
  - `QString controlStatus()` — calls `org.kde.FanControl.Control.get_control_status()`
  - `QString autoTuneResult(const QString &fanId)` — calls `org.kde.FanControl.Control.get_auto_tune_result(fanId)`
- Write methods (require root authorization per daemon policy):
  - `void setSensorName(const QString &id, const QString &name)` — calls Inventory.set_sensor_name
  - `void setFanName(const QString &id, const QString &name)` — calls Inventory.set_fan_name
  - `void removeSensorName(const QString &id)` — calls Inventory.remove_sensor_name
  - `void removeFanName(const QString &id)` — calls Inventory.remove_fan_name
  - `void setDraftFanEnrollment(const QString &fanId, const QString &draftJson)` — calls Lifecycle.set_draft_fan_enrollment
  - `void removeDraftFan(const QString &fanId)` — calls Lifecycle.remove_draft_fan
  - `void discardDraft()` — calls Lifecycle.discard_draft
  - `QString validateDraft()` — calls Lifecycle.validate_draft, returns JSON string
  - `QString applyDraft()` — calls Lifecycle.apply_draft, returns JSON string
  - `void startAutoTune(const QString &fanId)` — calls Control.start_auto_tune
  - `void acceptAutoTune(const QString &fanId)` — calls Control.accept_auto_tune
  - `void setDraftFanControlProfile(const QString &fanId, const QString &profileJson)` — calls Control.set_draft_fan_control_profile
- Error handling: catch QDBusError replies, emit `lastErrorChanged` with formatted message, set `connected` to `false` if daemon is unreachable

**4. StatusMonitor** (`gui/src/status_monitor.h` and `.cpp`):

- Q_PROPERTY: `bool daemonConnected` (based on name owner tracking on system bus)
- In constructor, connect DBus signals:
  - `org.kde.FanControl.Lifecycle.draft_changed` → call `refreshDraftConfig()`
  - `org.kde.FanControl.Lifecycle.applied_config_changed` → call `refreshAppliedConfig()`

- Each `refresh*()` method calls the corresponding DaemonInterface read method and updates the relevant model
- Use `QMetaObject::invokeMethod(this, ..., Qt::QueuedConnection)` to marshal DBus signal callbacks to the main thread

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GUI → DBus → Daemon | Untrusted GUI client communicates with privileged daemon over system bus. Daemon enforces authorization; GUI must surface errors. |
| DBus method errors | Daemon may return `fdo::Result::Err` for authorization or validation failures. GUI must display these to the user, never silently swallow. |
| Stale data | Daemon signals may arrive out of order or GUI may be disconnected. GUI must label stale data and reconnect gracefully. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | DBus write methods | mitigate | Daemon already requires UID 0 per `require_authorized()`. GUI must surface authorization errors to user per UI-SPEC contract ("This action requires elevated privileges..."). No client-side authorization pre-check — try and report errors. |
| T-04-02 | Information Disclosure | DBus read methods | accept | Per `org.kde.FanControl.conf`, all local users can read. No sensitive data beyond hardware state. |
| T-04-03 | Denial of Service | GUI → daemon connection | mitigate | If daemon is unreachable, GUI sets `connected = false` and shows disconnected banner per UI-SPEC. Write actions are disabled. Auto-reconnect on name owner change. |
| T-04-04 | Tampering | JSON parsing | mitigate | All daemon responses are JSON strings. C++ layer uses `QJsonDocument::fromJson()` with null/error checking. Malformed JSON results in error state, not crash. |
</threat_model>
