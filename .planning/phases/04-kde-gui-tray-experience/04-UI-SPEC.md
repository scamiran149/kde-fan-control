## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable — KDE desktop app, not shadcn |
| Component library | Kirigami 6 + Qt Quick Controls 2 |
| Icon library | KDE/Breeze system icon theme (`-symbolic` variants where available) |
| Font | KDE system sans-serif; default to platform UI font (typically Noto Sans) |

**Implementation shell**
- Use `Kirigami.ApplicationWindow` as the root shell.
- Use `pageStack` navigation: initial page is **Overview**; fan detail pushes onto the stack.
- Use `Kirigami.ScrollablePage` for Overview, Inventory, and Fan Detail.
- Use `Kirigami.Action` for all primary toolbar and row actions.
- Use `Kirigami.InlineMessage` for validation results, degraded summaries, daemon connection errors, and auto-tune proposal banners.
- Use `Kirigami.FormLayout` for editable fan settings.
- Use Qt Quick Controls `TabBar` + `StackLayout` or Kirigami tab pattern for advanced sections.
- Use `KStatusNotifierItem` for tray presence; tray remains inspect-first, not control-heavy.

## App Shell And Navigation

### Top-level pages
1. **Overview** — default landing page; operational dashboard.
2. **Inventory** — full discovered sensor/fan inventory and support reasons.
3. **Fan Detail** — pushed from Overview or Inventory for a selected fan.

### Window rules
- Minimum window size: **980x700**.
- Comfortable default desktop size: **1180x820**.
- Outer page padding: **24px**.
- Max readable form width in detail page: **720px**.
- At widths **>= 1200px**, Fan Detail may use a two-column layout:
  - left: runtime summary + core controls
  - right: advanced tabs / lifecycle history
- At widths **< 1200px**, collapse to one column; tabs move below core controls.

### Navigation rules
- Overview always opens first.
- Selecting a fan from Overview or Inventory opens Fan Detail.
- Do not expose a permanent settings-heavy sidebar by default.
- Optional global drawer/menu actions: `Overview`, `Inventory`, `About`, `Quit`.
- No profile switching, graphing, or future observability pages in v1.

### Visual focal point
The primary focal point on Overview is the top severity banner when a fallback, degraded, or disconnected banner is present. When no banner is active, the highest-urgency fan row serves as the visual anchor.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `Kirigami.Theme.backgroundColor` (Breeze Light reference `#fcfcfc`) | Window background, page background |
| Secondary (30%) | `Kirigami.Theme.alternateBackgroundColor` (Breeze Light reference `#eff0f1`) | Cards, grouped list rows, tray popover background |
| Accent (10%) | `Kirigami.Theme.highlightColor` (Breeze Light reference `#3daee9`) | Primary action emphasis, selected row highlight, focused output bar, wizard entry |
| Destructive | `Kirigami.Theme.negativeTextColor` (Breeze Light reference `#da4453`) | Fallback, destructive confirmations, irreversible warnings |

Accent reserved for: **Apply changes**, **Validate draft**, selected fan row, wizard entry action, accepted auto-tune proposal banner action, keyboard focus ring.

### State semantics
Use semantic KDE theme colors in addition to the 60/30/10 split:

| State | Color token | Icon | Label |
|---|---|---|---|
| Managed healthy | positive/success color | `emblem-ok-symbolic` | Managed |
| Unmanaged | neutral text color | `dialog-information-symbolic` | Unmanaged |
| Degraded | warning color | `data-warning-symbolic` | Degraded |
| Fallback | negative/destructive color | `dialog-error-symbolic` | Fallback |
| Partial support | warning color | `dialog-warning-symbolic` | Partial |
| Unsupported/unavailable | disabled/neutral color | `emblem-unavailable-symbolic` | Unsupported |
| High-temp alert | negative color accent pill layered on top of current state | `temperature-high-symbolic` | High temp |

**Rules**
- Color is never the only indicator; every state also needs icon + text.
- High-temp alert is a **separate alert pill**, not a replacement for the base runtime state.
- Unsupported rows use subdued surfaces; never use accent blue for unavailable hardware.

## Overview Contract

### Information hierarchy
Order rows by operational urgency:
1. fallback fans
2. degraded fans
3. managed fans with high-temp alert
4. managed healthy fans
5. unmanaged available fans
6. partial/unsupported fans

### Fan row/card content
Each fan entry must show:
- fan display name (`friendly_name`, else label, else stable channel name)
- state badge
- current input temperature:
  - managed fan: aggregated control temperature
  - unmanaged/unsupported: `No control source`
- RPM when present; `No RPM feedback` when absent
- current output bar:
  - managed fan: filled 0–100% with numeric percent label
  - unmanaged/degraded/fallback/unsupported: disabled or fixed semantic presentation, never fake a live value
- support reason as secondary text for partial/unsupported fans
- chevron/open action with accessible name `Open fan details` to navigate to Fan Detail

### Visual rules
- Use grouped list rows or compact cards; do not build dense tables.
- Output bar width: **96px minimum**, height **8px**, label aligned right.
- Secondary metadata is one line max on Overview.
- Overview favors live status over configuration text.

### Global banners
At top of Overview, show at most one banner per severity group:
- fallback banner
- degraded banner
- daemon disconnected banner
- auto-tune completion banner for currently selected fan only does **not** belong on Overview

## Inventory Contract

### Sensors section
Show all discovered temperature sensors with:
- display name
- current temperature
- source device name
- stable ID only in secondary or copyable detail view

### Fans section
Show all detected fans with:
- display name
- support state
- control modes available (`PWM`, `Voltage` when exposed)
- RPM feedback support
- support reason when partial/unavailable

### Rules
- Inventory is read-first and diagnostic.
- Stable IDs and sysfs-like internals are hidden by default and available only in secondary detail text or copy action.
- Do not duplicate the full editable control form here; detail editing happens on Fan Detail.

## Fan Detail Contract

### Header block
Always visible at top:
- fan name
- state badge
- managed/unmanaged support summary
- current aggregated temperature
- RPM
- output percent bar
- high-temp alert pill when active

### Core controls order
Show these before any advanced tab content:
1. Enrollment state toggle
2. Control mode selector
3. Temperature source selector
4. Aggregation selector
5. Target temperature field
6. PID gains: `Kp`, `Ki`, `Kd`
7. Inline `Start auto-tune` action

### Source selection rules
- Use a multi-select sensor list.
- If exactly one sensor is selected, replace aggregation dropdown with read-only text: `Single sensor`.
- Show aggregation dropdown only when **2 or more** sensors are selected.
- Aggregation choices are exactly: `Average`, `Max`, `Min`, `Median`.

### PID help text
Hover or info-tip copy:
- **Kp** — `Responds to the current temperature gap. Higher values react faster but can hunt.`
- **Ki** — `Corrects steady offset over time. Higher values remove drift but can overshoot.`
- **Kd** — `Damps fast temperature swings. Higher values can reduce overshoot but may amplify noise.`

### Advanced tabs
Use tabs, not accordions.

Required tabs:
1. **Runtime**
   - live sensor IDs/names used
   - aggregation mode
   - target temperature
   - current error
   - mapped PWM when present
   - auto-tune state
   - high-temp alert state
2. **Advanced**
   - cadence
   - deadband
   - actuator policy
   - PID limits
3. **Events**
   - degraded reasons
   - lifecycle event history with timestamp + reason + optional detail

### Form behavior
- Keep core controls in a single scannable pane above tabs.
- Advanced values are hidden from first glance.
- Use `°C` in UI; convert from millidegrees internally.
- Temperatures display with **1 decimal place**.
- Output displays as **whole percent**.
- RPM displays as integer.

## Draft Editing And Validation Contract

### Editing model
- Field changes are local in the detail pane until the user chooses `Validate draft` or `Apply changes`.
- `Validate draft` must first flush current local form values into daemon draft methods, then run DBus `validate_draft`.
- `Apply changes` must first flush current local form values into daemon draft methods, then run DBus `apply_draft`.
- `Discard draft` reverts the local pane to the last daemon draft state; if the user chooses full discard, call DBus `discard_draft`.

### Primary actions
In this order:
1. `Validate draft`
2. `Apply changes`
3. `Discard draft`

### Validation presentation
- Use a top inline message for overall result:
  - success: count of enrollable entries
  - warning/error: rejected fan count
- Show per-field inline error text for the active fan when validation rejects its values.
- Validation errors must use backend language, simplified for operators:
  - no target temperature
  - no temperature source
  - unsupported control mode
  - invalid cadence
  - invalid actuator policy
  - invalid PID limits
  - source or fan no longer found

### Apply result rules
- Best-effort partial apply must be surfaced explicitly.
- If apply rejects some fans, keep warning banner visible until dismissed or resolved.
- Never show a success toast if any fan was rejected.

## Wizard Contract

### Purpose
Optional guided path for first setup only; it does not replace direct editing.

### Entry points
- Empty state CTA
- Secondary action in Overview toolbar
- Secondary action in Fan Detail when fan is unmanaged

### Steps
1. Select fan
2. Select control mode
3. Select sensor source(s)
4. Choose aggregation if multi-sensor
5. Set target temperature
6. Review starter PID values
7. Review + `Validate draft` + `Apply changes`

### Rules
- Wizard uses the same backend draft/apply contract as direct editing.
- Wizard never bypasses validation.
- Wizard must end on a review screen with explicit apply.
- Advanced controls are not shown in the wizard.

## Auto-Tune Contract

### Entry and progress
- Auto-tune starts only from Fan Detail.
- Start action label: `Start auto-tune`.
- While running, show inline progress state in the header and Runtime tab.
- While auto-tune is running, disable a second start action for that fan.

### Completion
- On `AutoTuneCompleted`, show an inline proposal banner in Fan Detail:
  - heading: `Auto-tune finished`
  - body: `Review the proposed PID gains before applying them.`
  - actions: `Accept proposed gains`, `Dismiss proposal`
- `Accept proposed gains` stages the proposal into draft only.
- Accepted gains still require `Apply changes` to become live.

### Failure
- If auto-tune fails, show inline error:
  - `Auto-tune could not complete. Check sensor availability and current runtime state, then try again.`

## Tray Contract

### Purpose
Compact inspection and alert surface only.

### Popover structure
1. Header:
   - daemon connection state
   - worst active severity
   - counts: managed / alerts
   - primary action: `Open Fan Control`
2. Alert area:
   - sticky active alert summary if fallback, degraded, or high-temp exists
3. Managed fan list:
   - managed fans only by default
   - each row shows fan name, state icon, temperature, and output percent or RPM
4. Footer actions:
   - `Open Fan Control`
   - `Acknowledge alerts` only when sticky alerts exist

### Tray size rules
- Preferred width: **360px**
- Max visible managed fan rows before scrolling: **6**
- Row height: **40px minimum**

### Explicit exclusions
- No enrollment editing from tray
- No PID field editing from tray
- No mini control center sliders
- No profile switching

### Tray icon severity precedence
1. fallback
2. degraded
3. high-temp alert
4. managed healthy
5. no managed fans / unmanaged only
6. daemon unavailable

## Notifications Contract

### Trigger conditions
Only on transition into:
- degraded
- fallback
- high-temp alert

### Notification behavior
- Desktop popup may be transient.
- Alert must remain sticky in tray/main window until acknowledged by the user.
- Acknowledging clears UI stickiness only; it does not alter daemon state.

### Notification copy
- **Degraded:** `Fan control degraded` — `One or more managed fans could not be controlled safely. Open Fan Control to review the reason.`
- **Fallback:** `Fallback active` — `Managed fans were driven to safe maximum output. Open Fan Control now.`
- **High temp:** `High temperature alert` — `A managed fan is above its target temperature. Check runtime status and tuning.`

---

## Connection And Authorization Contract

### DBus connection states
Show one of:
- `Connected`
- `Connecting`
- `Disconnected`

### Disconnected behavior
- Keep last-known values visually stale only if labeled `Last known`.
- Disable write actions.
- Show inline error banner:
  - `Couldn't talk to the fan-control daemon. Check that the system service is running, then retry.`

### Authorization behavior
- Read views remain available to local users.
- If a privileged DBus write fails authorization, show inline error:
  - `This action requires elevated privileges. Run the GUI with the required authorization and try again.`
- Never pretend a write succeeded when authorization failed.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Apply changes |
| Empty state heading | No managed fans yet |
| Empty state body | Select a supported fan, choose its temperature source, then validate and apply the draft to start daemon-managed control. |
| Error state | Couldn't talk to the fan-control daemon. Check that the system service is running, then retry. |
| Destructive confirmation | Discard draft changes: `Discard all staged changes? This keeps the current applied configuration and removes the current draft.`; Stop managing fan: `Stop daemon control for this fan on next apply? The fan will return to unmanaged state.` |

### Key labels
- `Validate draft`
- `Apply changes`
- `Discard draft`
- `Wizard configuration`
- `Start auto-tune`
- `Accept proposed gains`
- `Dismiss proposal`
- `Open fan details`
- `Open Fan Control`
- `Acknowledge alerts`

## Accessibility And Motion

- Minimum interactive control height: **32px** in main window; **40px** in tray rows.
- Every state indicator must include text, icon, and color.
- All actions reachable by keyboard; tab order follows visual order.
- Focus ring uses accent color and **2px** visible outline.
- Tooltips appear on hover and keyboard focus for PID help affordances.
- Motion duration: **120–160ms** for page/stack transitions.
- No continuous animation except explicit auto-tune running indicator.
- Do not rely on hover-only affordances for critical actions; tooltip helpers supplement visible labels.

## Data Binding Contract

Bind only to existing daemon/client surfaces:

### Read
- `org.kde.FanControl.Inventory`
- `org.kde.FanControl.Lifecycle.get_draft_config`
- `org.kde.FanControl.Lifecycle.get_applied_config`
- `org.kde.FanControl.Lifecycle.get_degraded_summary`
- `org.kde.FanControl.Lifecycle.get_lifecycle_events`
- `org.kde.FanControl.Lifecycle.get_runtime_state`
- `org.kde.FanControl.Control.get_control_status`
- `org.kde.FanControl.Control.get_auto_tune_result`

### Write
- fan enrollment draft methods
- fan control profile draft methods
- `validate_draft`
- `apply_draft`
- `discard_draft`
- `start_auto_tune`
- `accept_auto_tune`

### Reactive updates
Subscribe to:
- `draft_changed`
- `degraded_state_changed`
- `control_status_changed`
- `AutoTuneCompleted`

No direct sysfs access. No secondary config authority. No speculative backend-only features.
