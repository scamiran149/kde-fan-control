  <name>Task 1: Add sensor reading, live sysfs path resolution, auto-tune step response, and ControlStatus to the control module</name>
  <files>crates/core/src/control.rs</files>
  <action>
Extend `crates/core/src/control.rs` with:

1. **Sensor reading function**: `pub fn read_sensor_millidegrees(sysfs_path: &str, channel: u32) -> io::Result<i64>` that reads from `{sysfs_path}/temp{channel}_input`, parses the string as i64, returns the millidegree value.

2. **Sensor aggregation with live reading**: `pub fn aggregate_sensor_readings(sensor_paths: &[(String, u32)], aggregation_fn: AggregationFn) -> Result<i64, String>` that:
   - Reads each sensor using `read_sensor_millidegrees`
   - Filters out failed reads (log warnings but don't fail the whole group)
   - If ALL sensors fail, return Err
   - Applies the aggregation function to convert readings to a single millidegree value
   - For a single sensor, skip aggregation (just return the reading)

3. **ControlStatus per-fan struct**: `pub struct FanControlStatus` with fields:
   - `fan_id: String`
   - `control_mode: ControlMode`
   - `target_temp_c: f64` (degrees C, for display convenience)
   - `aggregated_temp_c: Option<f64>`
   - `pid_output: Option<PidOutput>`
   - `pwm_value: Option<u8>`
   - `auto_tune_state: AutoTuneState`
   - `tick_interval_ms: u64`

4. **Ziegler-Nichols step-response auto-tune function**: `pub fn compute_auto_tune_gains(lag_time_ms: u64, max_rate_per_sec: f64) -> PidGains`:
   - Kp = 1.2 / (R * L) where R = max_rate_per_sec (degrees per second), L = lag_time_ms / 1000.0 (seconds)
   - Ki = Kp / (2.0 * L)
   - Kd = Kp * L / 0.5
   - If lag_time_ms is 0 or max_rate is 0, return default gains (same as config defaults)

5. **Add unit tests**:
   - `read_sensor_parses_millidegrees`: use a temp file as fake sysfs, write "45000\n", read back as 45000
   - `aggregate_sensor_readings_average`: multiple readings, average computation
   - `aggregate_sensor_readings_all_fail`: returns error when all sensors fail
   - `aggregate_sensor_readings_partial_fail`: continues with remaining sensors
   - `compute_auto_tune_gains_known_input`: verify ZN formulas with known lag and rate
   - `compute_auto_tune_gains_zero_values`: returns default gains when inputs are degenerate

<task type="auto">
  <name>Task 2: Add DBus Control interface, control supervisor, per-fan PID tasks, and auto-tune trigger to the daemon</name>
  <files>crates/daemon/src/main.rs</files>
  <action>
Add the Control DBus interface and control task architecture to `crates/daemon/src/main.rs`:

1. **FanControlState struct** (per-fan runtime state held behind Arc<RwLock>):
   - `pid_controller: Option<PidController>` — active PID for this fan
   - `aggregation_fn: AggregationFn`
   - `sensor_paths: Vec<(String, u32)>` — (device_sysfs_path, channel) tuples for live reading
   - `tick_interval_ms: u64`
   - `last_output: Option<PidOutput>`
   - `auto_tune_state: AutoTuneState`
   - `cancellation: tokio::sync::watch::Sender<bool>` — for graceful task shutdown per fan

2. **ControlSupervisor struct** managing per-fan tasks:
   - `tasks: HashMap<String, tokio::task::JoinHandle<()>>` — running PID tasks
   - `states: Arc<RwLock<HashMap<String, FanControlState>>>` — per-fan runtime state
   - `config: Arc<RwLock<AppConfig>>`, `owned: Arc<RwLock<OwnedFanSet>>`, `snapshot: Arc<RwLock<InventorySnapshot>>`
   - Method `start_fan_control(fan_id)` — spawns PID task for a fan
   - Method `stop_fan_control(fan_id)` — signals cancellation and joins task
   - Method `reconcile_from_applied_config()` — starts tasks for all managed fans in applied config, stops tasks for no-longer-managed fans
   - Method `get_control_statuses()` — returns current FanControlStatus for all managed fans

3. **Per-fan PID task** (spawned via `tokio::spawn`):
   - Uses `tokio::time::interval(Duration::from_millis(tick_interval))` for periodic ticks
   - On each tick:
     a. Read live sensor values using `aggregate_sensor_readings(sensor_paths, aggregation_fn)`
     b. If ALL sensors fail: mark fan as degraded (DegradedReason::TempSourceMissing), stop writing PWM, update runtime status
     c. Compute PID output via `pid_controller.compute(aggregated_millidegrees)`
     d. Write PWM value (clamp output to u8 0-255) to the fan's sysfs PWM path
     e. Update FanControlState with last_output
   - Listen for cancellation via `tokio::sync::watch` receiver
   - On cancellation or degraded transition: gracefully exit

4. **ControlIface DBus interface** at path `/org/kde/FanControl/Control`:
   - `get_control_status() -> String` (JSON) — returns FanControlStatus for all managed fans
   - `start_auto_tune(fan_id: String) -> String` (JSON) — starts auto-tune for a fan (privileged)
   - `get_auto_tune_result(fan_id: String) -> String` (JSON) — returns auto-tune result for a fan
   - Accept the fan's proposed tuned gains into the draft config (privileged)
   - Signal: `ControlStatusChanged()` — emitted when control status updates significantly
   - Signal: `AutoTuneCompleted(fan_id, proposed_kp, proposed_ki, proposed_kd)` — emitted when auto-tune finishes

5. **Auto-tune within PID task**:
   - When `start_auto_tune` is called for a fan, set the fan's auto_tune_state to Running
   - In the PID task, when auto_tune_state is Running:
     a. Phase 1 (observation): Run the fan at max PWM (255) for a step-duration (e.g., 30 seconds)
     b. Monitor the temperature response: track when the temperature starts dropping (lag time L) and the max rate of change (R)
     c. Compute proposed gains via `compute_auto_tune_gains(L, R)`
     d. Transition to Completed state with the proposed gains
   - The fan continues running at max during auto-tune, then reverts to normal PID control with its existing gains
   - The proposed gains are stored in AutoTuneState::Completed and exposed via DBus for user review
   - The user must explicitly accept the proposed gains through a separate draft/apply cycle

6. **Integration in main()**:
   - Create and register ControlIface with shared state references
   - After boot reconciliation, call `control_supervisor.reconcile_from_applied_config()`
   - On apply_draft via Lifecycle DBus: after updating applied config and owned set, call `control_supervisor.reconcile_from_applied_config()` to start/stop PID tasks as needed
   - On shutdown: the existing fallback logic drives fans to max before exit — no need to gracefully stop PID tasks first (they'll be killed)

7. **Extend FanRuntimeStatus** in lifecycle.rs: add fields to `Managed` variant:
   - Change `Managed { control_mode: ControlMode }` to `Managed { control_mode: ControlMode, auto_tuning: bool }`
   - This extends the existing enum — update all match arms

Add imports for `kde_fan_control_core::control::*` in main.rs.

Include tests:
- `control_status_serializes_correctly`: FanControlStatus serializes to JSON
- `auto_tune_computes_reasonable_gains`: known lag and rate produce non-zero gains
- `pid_task_reads_and_writes`: test the PID tick computation path with mock sysfs data

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DBus client → daemon Control interface | Untrusted callers can request auto-tune (privileged) or read status (unprivileged) |
| PID task → sysfs PWM write | Daemon writes to hardware files; output must be clamped to [0, 255] |
| PID task → sysfs temp read | Daemon reads live sensor values; must handle read failures gracefully |
