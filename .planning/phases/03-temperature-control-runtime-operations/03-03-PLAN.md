  <files>crates/cli/src/main.rs</files>
  <action>
Extend `crates/cli/src/main.rs`:

1. **Add ControlProxy** — a `#[proxy]` trait matching the `org.kde.FanControl.Control` DBus interface:
   - `fn get_control_status(&self) -> zbus::Result<String>`
   - `fn start_auto_tune(&self, fan_id: &str) -> zbus::Result<String>`
   - `fn get_auto_tune_result(&self, fan_id: &str) -> zbus::Result<String>`
   - `fn accept_auto_tune(&self, fan_id: &str) -> zbus::Result<String>`
   - Path: `/org/kde/FanControl/Control`, Service: `org.kde.FanControl`

2. **Add `connect_control_proxy()` async helper** following the pattern of `connect_lifecycle_proxy()`.

3. **Add CLI subcommands**:
   - `AutoTune { fan_id: String }` — starts auto-tuning for a specific fan
   - `TuneResult { fan_id: String }` — shows the auto-tune result (proposed gains) for a fan
   - `TuneAccept { fan_id: String }` — accepts the proposed auto-tune gains and stages them in the draft
   
4. **Extend existing `Enroll` command** with additional options:
   - `--target-temp <f64>` — target temperature in degrees Celsius (stored as millidegrees internally: multiply by 1000.0 and cast to i64)
   - `--kp <f64>` — proportional gain (optional, uses default if omitted)
   - `--ki <f64>` — integral gain (optional, uses default if omitted)
   - `--kd <f64>` — derivative gain (optional, uses default if omitted)
   - `--aggregation <str>` — aggregation function: "average" | "max" | "min" | "median" (default: average)
   - `--tick-interval <u64>` — tick interval in milliseconds (default: 2000)
   These are sent to the existing `set_draft_fan_enrollment` DBus method — the daemon's DBus method signature may need updating to accept these new fields. If so, extend the `set_draft_fan_enrollment` method in the daemon to accept the additional parameters (target_temp, kp, ki, kd, aggregation, tick_interval) and store them in the DraftFanEntry.

5. **Enhance `State` command output**: after the existing lifecycle status, add a "CONTROL STATUS" section that:
   - Calls `get_control_status()` on the Control interface
   - For each managed fan, displays:
     - Target temperature (°C)
     - Current aggregated temperature (°C) if available
     - PWM output value (0-255)
     - PID error (°C)
     - Auto-tune state (Idle / Running / Completed with proposed gains)
   - Clear v1 label: "Note: v1 control is temperature-target PID (fan speed tracks a target temperature), not RPM-target tracking."

6. **Implement auto-tune UX flow**:
   - `auto-tune <fan_id>`: Calls `start_auto_tune()`, prints "Auto-tuning started for {fan_id}. This will run for approximately 30 seconds. Use 'tune-result {fan_id}' to check progress and results."
   - `tune-result <fan_id>`: Calls `get_auto_tune_result()`, prints:
     - If idle: "No auto-tune result available for {fan_id}."
     - If running: "Auto-tuning is still in progress for {fan_id}."
     - If completed: "Auto-tune completed for {fan_id}: Kp={proposed_kp}, Ki={proposed_ki}, Kd={proposed_kd}. Use 'tune-accept {fan_id}' to stage these values in the draft."
   - `tune-accept <fan_id>`: Calls `accept_auto_tune()`, prints "Proposed gains for {fan_id} staged in the draft configuration. Use 'apply' to make them live."

7. **Handle the `set_draft_fan_enrollment` signature change**: If extending the DBus method to accept PID parameters, update the existing `LifecycleProxy` trait and the `Enroll` command handler to pass the new arguments. The approach is: add optional parameters to the existing DBus call or add a separate `set_draft_fan_pid_params` method. Recommended: add a new method `set_draft_fan_pid_params(fan_id, target_temp_millidegrees, kp, ki, kd, aggregation, tick_interval_ms)` on the Lifecycle interface. This keeps the enrollment signature stable and lets the CLI call both methods for full fan setup.

Include the v1 PID mode note in both the `state` output and the `enroll` help text: "v1 uses temperature-target PID control — fans track a target temperature, not an RPM target."
